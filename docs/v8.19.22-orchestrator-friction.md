# Orchestrator UX Friction (v8.19.22)

What a fresh Claude Code session — asked to drive a worker via the
documented `culture boss …` flow — has to figure out that it
shouldn't have to. Each entry is a real moment of confusion from
this session, captured so the team can decide whether to fix the
gap or the docs.

## 1. `culture agent status` floods on stale worktree refs

Every invocation prints a stack of `WARNING culture.yaml missing for
<nick> at /private/tmp/<worktree>-tree — skipping`. The manifest is
holding pointers to worktrees that were `git worktree remove`'d
sessions ago. The lines are noise on every command, including
`boss status`, `boss audit`, `agent status`.

**Fix options:** auto-prune on `manifest` load when target is gone;
or surface ONE summary line ("8 manifest entries point to deleted
paths — run `culture agent forget <nick>` to clean up").

## 2. `culture boss spawn <name> --cwd <repo>` errors on multi-agent cwd

When the target cwd already has a `culture.yaml` with multiple
agents, spawn prints `Multiple agents in <path>. Use --suffix to
specify which one.` then continues anyway. Confusing — looks like
an error but the worker DOES come up.

**Fix:** silent default-to-`<name>` when --suffix is omitted AND the
name matches a defined suffix; or hard-fail with clear next step.

## 3. `culture boss brief` fails with "boss daemon running?" — no fix path

```
$ culture boss brief ux-w "<task>"
Error: could not brief local-ux-w (is the boss daemon running?)
```

True, but the orchestrator now has to know:

- That a boss **daemon** is a separate process from the boss
  **identity**.
- That `culture boss status` doesn't tell them the boss daemon is
  down (it lists workers, not the boss's own daemon).
- That the fix is `culture agent start local-boss` — never stated.

**Fix:** the error message should print the exact command to run.

## 4. `culture boss status` doesn't show the BOSS's own state

It shows workers. The boss's own daemon (the thing brief depends
on) is invisible. New orchestrators learn this by failure.

**Fix:** print a top line `BOSS local-boss [running|stopped] PID N`
before the worker table.

## 5. No "I am the orchestrator" entry-point

There's no `culture orchestrate` or `culture boss init` that
explains: *you are about to drive workers from outside the mesh,
here's what you need running (boss daemon + IRC server), here's how
to confirm, here are the verbs.*

**Fix:** a one-shot `culture boss init` that:

- Ensures the IRC server is up (`culture server status`).
- Ensures the boss daemon is up (`culture agent start <self>` if
  missing).
- Prints a confirmation banner.

## 6. New workers don't get onboarded — they get HISTORY

When a worker spawns into `#task-<name>`, it joins the channel,
reads the last N HISTORY lines, and that's its entire context for
"what is this work?" The v8.19.18 seed brief is write-once initial
brief — it never updates as the work progresses.

In a real team, a new joiner gets a *living* project doc — current
state, recent decisions, what's done, what's stuck, who's doing
what. A new mesh worker gets none of that.

**Proposed:** `~/.culture/briefs/<channel>.md` — a living per-channel
brief that the boss/orchestrator updates as work progresses. Format:

```
# Channel: #task-feature-x

## Mission
<from the seed, or the boss's most recent brief>

## Current state (auto-updated)
- Started: 2026-05-30T12:00
- Last activity: 2026-05-31T09:14
- Cumulative tokens: 487k
- Open threads: ...
- Recently completed: ...

## Onboarding for new joiners
<the orchestrator's running notes for anyone joining the room>
```

When a new worker spawns into a channel that has a brief, the
daemon injects the brief into the SDK system prompt (similar to
how the v8.19.3 mission persistence injects the boss's mission).
This makes the mesh actually onboard new joiners rather than just
dropping them into a chat log.

Separately, `culture boss brief` (or an explicit
`culture boss brief --append`) could write to the living brief
alongside posting the IRC message — so the channel brief
auto-accumulates without the orchestrator having to edit it.

**Open design question:** does the brief replace the seed
(extending the v8.19.18 file with append semantics) or is it a
separate channel-state file? Argument for replace: one source of
truth. Argument for separate: keeps the immutable original mission
visible alongside the evolving state.

## Why this matters

The mesh exists so an outside Claude Code session can drive a team
without becoming a developer of the mesh itself. Every friction
above is a moment where the orchestrator has to abandon their
actual task (drive workers) to debug the harness. The Pattern A
vision (human → orchestrator → mesh) only works if the orchestrator
can succeed on the documented path.

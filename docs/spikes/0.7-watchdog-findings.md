# Spike 0.7 — watchdog + asyncio integration findings

**Date:** 2026-06-03
**Library:** `watchdog==6.0.0`
**Platform:** Darwin 25.5.0 (macOS, arm64)
**Observer backend selected:** `watchdog.observers.fsevents.FSEventsObserver`
**Script:** `docs/spikes/0.7-watchdog-smoke.py`
**Validates:** Phase 5.4 + 5.6 — perm-broker FS observer replacement.

## Result

PASS on every run. The `loop.call_soon_threadsafe` handoff from watchdog's
threaded `Observer` into the asyncio loop works exactly as the Phase 5 plan
assumes — `on_created` fires on watchdog's internal thread, schedules
`Future.set_result` on the main loop, and the awaiting coroutine resumes.

## Latency (5 consecutive runs, single-file create)

| Run | Latency (write start to future.set_result) |
| --- | ------------------------------------------ |
| 1   | 13.9 ms                                    |
| 2   | 14.9 ms                                    |
| 3   | 12.5 ms                                    |
| 4   | 13.0 ms                                    |
| 5   |  7.9 ms                                    |

Initial warm-up run (separate invocation) measured 20.7 ms. All runs were
well under the 5 s timeout — the famous "FSEvents 1-2 s coalesce" stall
did NOT manifest on `/tmp` (APFS) for single-file `write_text` followed by
implicit close. That stall mainly hits external/network volumes and very
high event rates; the perm-broker's grant-file directory will look like
this benchmark in practice.

## Platform notes

- `Observer()` on Darwin auto-selects `FSEventsObserver` (native CoreFoundation
  FSEvents stream). No kqueue / polling fallback was needed.
- Observer thread is daemon by default; `observer.stop(); observer.join(timeout)`
  cleanly terminates without leaking the FSEvents stream.
- A short `await asyncio.sleep(0.1)` after `observer.start()` is needed
  before the first write — FSEvents needs a beat to attach the stream.
  Without it, very fast writes (sub-millisecond after `start()`) can be
  missed because the stream is not yet active. Production code should also
  do a post-start settle, or accept that events in the first ~50 ms after
  `start()` may be lost.
- `PatternMatchingEventHandler(patterns=["*.json"])` correctly filtered out
  the directory-create event from `mkdir`; only the `*.json` `on_created`
  fired.

## Conclusion

The pattern is sound for Phase 5.4 + 5.6. The perm-broker can replace its
current polling loop with a watchdog `Observer` and a single `asyncio.Queue`
fed from `loop.call_soon_threadsafe` callbacks, with no expected latency
penalty versus polling (in fact, an improvement — polling typically pays
its full interval in worst-case, here we see ~10-20 ms event-to-coroutine).
No additional adapter library (e.g. `aiowatchdog`) is required.

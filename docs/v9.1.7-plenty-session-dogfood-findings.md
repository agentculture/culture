# v9.1.7 dogfood findings — Plenty ship-night session (2026-06-05)

In-mesh dogfood from a real Claude Code ship-night session on the **plenty.ai**
project (boss `plenty-ai-guide-mobile`). The mesh was reached for during a
`develop → main` production release to parallelize verification work. This doc
captures what worked and two open bugs. Supersedes the ad-hoc bug note that was
DM'd toward `@mesh-dev` (the mesh itself was wedged — Bug 1 — so it couldn't be
delivered in-mesh; filing here instead since the repo has GitHub issues disabled).

Boss = `plenty-ai-guide-mobile`. Mesh ≈ v9.1.7. Full timestamps/logs available on request.

## What worked

- **rc.1 checkpointing, `grant_ceiling` removal, and the polling-strip** all
  functioned as intended.
- **9.1.7's drift handling fired correctly.** The `432 ERR_ERRONEUSNICKNAME`
  now carries actionable reason text (`Nickname must start with local-` +
  "run `culture server migrate-prefix <old> <new>`") instead of the pre-9.1.6
  silent timeout / pre-9.1.7 infinite read-loop. That diagnostic is exactly
  what let me root-cause Bug 1 in minutes rather than hours. Credit where due —
  see [`server-name-drift-recovery.md`](server-name-drift-recovery.md).

The two findings below are **prevention / lifecycle** gaps, not detection gaps.

## Bug 1 (CRITICAL) — `server.name` drift keeps being *introduced* by writers the recovery doc doesn't cover

`docs/server-name-drift-recovery.md` handles drift **once it occurs** (observer
auto-retry; bridge fail-loud → daemon exit 1) and lists its causes as *operator
edited `server.yaml` in place* or *`culture server rename` without restart*.
This session hit **two automated writers not in that list**, so drift kept
re-appearing on its own:

1. **The bridge derives `server.name` from the session cwd basename and never
   reconciles it against the running IRCd's `--name`.** Onboarding this CC
   session as boss `plenty-ai-guide-mobile` left `~/.culture/server.yaml::server.name`
   = `plenty`, while the long-running IRCd was started with `--name local`.
   Every worker/observer the bridge then spawned registered as `plenty-*` /
   `plenty-_peek<rand>` and was 432-rejected.
2. **A pytest wrote into the real `~/.culture/server.yaml`.**
   `test_spawn_at_exact_limit` clobbered `server.name` to a 30-char fixture
   string and injected a 31-char agent entry at a `/private/tmp/.../pytest-*`
   path. (Manually reset to `local` at the time.)

Net effect: `server.name` churned `local → <fixture> → local → plenty` within a
single session. Any value ≠ the IRCd's `--name` hard-fails **all new
registrations** — `boss brief <w>` and worker channel-joins both 432.

**Impact:** `agent spawn` + perm-patch succeed, but `boss brief` / channel-join
always fail → **no work can be dispatched → mesh effectively down for new
tasks.** (9.1.7's fail-loud correctly *surfaces* this; the point is the drift
should never have been introduced.)

**Verbatim error:**
```
could not verify #task-meshcheck membership (IRC server rejected observer
registration: 432 'Nickname must start with local-'. Attempted nick
'plenty-_peek9581' against localhost:6667 with server_name='plenty'
(read from server.yaml). ... run `culture server migrate-prefix <old> <new>` ...)
```

**Proposed fix:**
- Bridge onboarding must **read the running IRCd's `--name`** and register under
  that prefix — or **refuse to onboard on mismatch** with a clear message —
  instead of silently deriving `server.name` from the cwd basename.
- **Isolate the test config dir** (honor `CULTURE_HOME` / a fixture tmpdir per
  `docs/culture-home-env.md`) so no test can write the real
  `~/.culture/server.yaml`. Verify the spawn-limit tests actually respect it.
- Operator short-term recovery already documented:
  `culture server migrate-prefix plenty local` + restart the IRCd with matching `--name`.

## Bug 2 (HIGH) — archived-worker ownership severed by `archive → unarchive/restore`

Workers spawned by an old boss → `agent archive` → later `agent unarchive` +
`agent start` come back **RUNNING but unmanageable by any boss**:

- `boss read` / `boss brief` → `REFUSED: <w> is not your worker (owned by another boss)`
- `agent stop` → `REFUSED: not your child — only its parent may close it`

This fires **even after** `agent restore <old-boss>` + start — both the new
bridge boss *and* the restored original owner are refused. Only `kill <pid>`
clears the orphaned processes.

**Likely cause:** unarchive/restore doesn't re-link `boss_owner`, and/or the
`plenty-` vs `local-` server-prefix split (Bug 1) breaks the ownership-match key.

**Proposed fix:** re-establish `boss_owner` on unarchive/restore (and verify the
ownership match isn't keyed on the drifted server prefix).

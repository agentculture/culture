# v9.1.9 — E2E harness + IRCd ACL isolation fix

This release lands a real end-to-end test harness for the boss/worker
flow and fixes the load-bearing isolation defect the harness surfaced
on its first run.

## What was actually wrong

Between v9.1.5 and v9.1.8 the project shipped four PRs that *should*
have closed "no session has gotten past brief" (the operational claim
captured in [`v9.1.7-plenty-session-dogfood-findings.md`](v9.1.7-plenty-session-dogfood-findings.md)).
None of them did, because all four were verified by unit tests that
never exercised the actual failure path:

```text
culture/agentirc/client.py::_load_owner_map
└── reads server.yaml at culture_home()
    └── culture_home() reads os.environ["CULTURE_HOME"]
        └── pytest never set this on its own process
            └── IRCd reads the OPERATOR's REAL ~/.culture/server.yaml
                └── test workers aren't listed in the operator's
                    real manifest → IRCd 432-rejects every JOIN to
                    #task-<suffix>
```

A first-pass e2e harness pinned the failure end-to-end (raw IRCd in
pytest's process + real ``culture bridge`` subprocess + fake-worker
TCP client + real ``culture boss brief`` subprocess) and traced
the rejection back to the env leak above.

## The fix

`tests/e2e/conftest.py` — the `e2e_ircd` fixture now binds
``CULTURE_HOME`` (and the symmetric ``HOME`` / ``XDG_RUNTIME_DIR``)
on the **pytest process** via ``monkeypatch.setenv``, not just on
the subprocess env (which `e2e_env` already covered). The module-level
owner_map / role_map caches in `culture/agentirc/client.py` are also
flushed at fixture entry AND exit so prior-test state cannot leak into
the next test, and vice versa.

`register_fake_worker` now also invalidates the ACL cache after
mutating the manifest, because the cache TTL is 5 seconds and is keyed
on path (not mtime) — without an explicit invalidation, an ACL check
fired within 5s of the manifest write would see stale "no such worker"
data and reject the JOIN.

## E2E test coverage shipped

- `test_brief_happy_path.py` — full brief delivery: in-process IRCd →
  real bridge subprocess → fake worker on `#task-qa` receives the
  PRIVMSG on the wire. **The load-bearing test.**
- `test_multi_worker_isolation.py` — brief to worker `qa` never appears
  on worker `docs`'s `#task-docs`. Pins the no-cross-talk guarantee.
- `test_foreign_boss_refused.py` (×2) — bosses cannot brief workers
  owned by another boss; orphan workers (not in manifest) are foreign
  to every boss (fail closed).
- `test_archive_unarchive_ownership.py` — Plenty BUG 2 reproduction
  attempt. With a CONSISTENT server prefix, archive → unarchive
  preserves `boss:` ownership and brief still works. This validates
  that BUG 2 was downstream of BUG 1 (server-prefix drift), not an
  independent unarchive defect.
- `test_server_name_drift.py` (×2) — Plenty BUG 1. Bridge subprocess
  fails loud on drifted `server.name` (v9.1.7 contract), and the
  matching-name case continues to work.

All seven tests pass. The real `~/.culture/server.yaml` is never read
or written during the suite (md5 verified pre/post).

## Bonus: brief CLI error string clarified

`culture/cli/boss.py::_cmd_brief` no longer collapses "IPC connect
failed" and "IPC connected but the bridge replied with an error" into
a single misleading "the boss daemon is not reachable over IPC"
message. The new error surfaces the bridge's actual `error` text,
which usually exposes the underlying timing race (`Not joined to
<channel>`) instead of pointing the operator at the wrong remediation.

## How to extend the harness

`tests/e2e/conftest.py` provides:

- `e2e_root` — fresh sandbox under `/tmp/cu-<uuid>`
- `e2e_env` — subprocess env (CULTURE_HOME / HOME / XDG_RUNTIME_DIR set)
- `e2e_ircd` — in-process IRCd on a random port (sandboxed)
- `seed_server_yaml(name, host, port)` — write `server.yaml`
- `register_fake_worker(suffix, boss_nick)` — manifest + culture.yaml
- `bridge(nick)` — real `culture bridge` subprocess, waited until on the mesh
- `bridge_join(boss_nick, channel)` — IPC `irc_join` + wait for JOIN echo
- `fake_worker(nick, channel)` — raw IRC TCP client on the mesh
- `boss_cli(*args, nick=...)` — async `python -m culture boss <args>`
- `agent_cli(*args)` — sync `python -m culture agent <args>`

Write your test as `async def` with `@pytest.mark.asyncio`, request the
fixtures you need, and assert against the wire (`fake_worker`'s
`received_privmsgs()`). No mocks; everything runs end-to-end.
